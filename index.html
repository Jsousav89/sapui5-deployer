SAPUI5-Deployer

ABAP deployment of SAPUI5 apps from Github

View project on GitHub

Welcome to SAPUI5-Deployer

SAPUI5-Deployer is a solution for deploying SAPUI5 and OpenUI5 web apps to a SAP ABAP Web Application Server automatically. It is designed to simplify the developer workflow by automating the following steps.

  • Download of Github Archive
  • Remove wrapper subdirectory from archive
  • Minify CSS, Javascript & XML files
  • Build Component-preload.js file where required
  • Recursively perform above steps for all Github submodules
  • Build complete new archive ready for deployment
  • Save deployment archive to file system
  • Deploy to SAP Internet Communications Framework
  • Flush cache

System Requirements

This solution was built on a SAP WAS ABAP 7.31 SP5 system with the SAP_UI software component also installed. There is no reason it should not work on all releases that have the function module /ui5/ui5_repository_load_http.

The installation instructions include making modifications to this function module to support the direct input of the Github archive. It is possible that SAP will modify this function module to support this in a future release.

Uwe Fetzer's (@se38) zJSON project for deserialising JSON into ABAP.

GitHub requires HTTPS access so the ABAP ICF will need to have this SSL setup and the appropriate certificates installed using transaction STRUST.

Installation

SSL setup

Refer to the relevant SAP Documentation Configuring the SAP Web AS for Supporting SSL.

As you will only be using this code on a development system you may choose to use a self-signed certificate.

Download the CA root certificate(s) that GitHub uses from https://www.digicert.com/digicert-root-certificates.htm. You can find which specific certificate GitHub uses from the GitHub documentation or use your web browser to determine it.

Install the certificate(s) in STRUST under "SSL System Client SSL Client (Anonymous)"

Restart ICM using transaction SMICM

Function module /ui5/ui5_repository_load_http

The main functionality used to deploy the archive is in Function Module /UI5/UI5_REPOSITORY_LOAD_HTTP that is part of the Software Component SAP_UI. You can easily figure out how to use this function module by looking at report /UI5/UI5_REPOSITORY_LOAD_HTTP. *Yes the FM and the Report have the same name ;)

In this Report you can also find the syntax of the parameters file that can be used to provide the details like Name of Application etc.

We need to modify this function module slightly as the original version reads the zip archive to be processed via http-call while we want to hand it over as binary data (XSTRING).

You can choose to modify the SAP delivered function module or make your own copy for the purpose.

Make the following modifications to the Function Module /UI5/UI5_REPOSITORY_LOAD_HTTP

Add importing parameter IV_ZIP_FILE (optional and pass value):

IV_ZIP_FILE TYPE XSTRING  The zip archive

Hand over the zip file to the local class (around line 171 of FM /UI5/UI5_REPOSITORY_LOAD_HTTP):

GET REFERENCE OF ev_log_messages INTO log_messages_ref.
CREATE OBJECT sapui5_archive
EXPORTING
  iv_url = iv_url
*{ INSERT IFDK944063 1
  iv_zip_file = iv_zip_file
*} INSERT
  ir_log_messages = log_messages_ref.

Add the importing parameter iv_zip_file to the local class definition for lcl_sapui5_zip_archive. This is around line 136 of the /UI5/LUI5_REPOSITORY_LOADD01 include

methods:   constructor  importing iv_url type string
                                  iv_zip_file type xstring optional
                                  ir_log_messages type ref to string_table,

In the implementation of this class (include /UI5/LUI5_REPOSITORY_LOADP01) around line 517

*{ INSERT IFDK944085 1
if iv_zip_file is not supplied.
*} INSERT

...and at line approx 556

DATA: xstring TYPE xstring.
  xstring = http_client->response->get_data( ).
*{ INSERT IFDK944085 2
else.
  xstring = iv_zip_file.
endif.
*} INSERT

YCL_GITHUB_DEPLOYER Class

This code makes use of the zJSON project from Uwe Fetzer (@se38). Follow the project instructions to install.

Create class YCL_GITHUB_DEPLOYER and then copy and paste in the code from the repo. There is also a SAPLink nugget containing the code if you prefer that.

How To Use

A sample program that calls the YCL_GITHIB_DEPLOYER class. Note that github user credentials are only required for private repos.

REPORT ygithub_deploy_ui5_repository.

DATA: exc TYPE REF TO cx_root.

SELECTION-SCREEN BEGIN OF BLOCK blk_main.
PARAMETERS:
  git_repo TYPE string OBLIGATORY LOWER CASE
    DEFAULT `Yelcho/openui5-project`,
  branch TYPE string OBLIGATORY LOWER CASE
    DEFAULT `master`,
  username TYPE string VISIBLE LENGTH 15 LOWER CASE DEFAULT 'mygituser',
  password TYPE string VISIBLE LENGTH 15 LOWER CASE,
  tranport TYPE trkorr DEFAULT 'DEVK900001',
  test TYPE boolean DEFAULT 'X'.
SELECTION-SCREEN END OF BLOCK blk_main.

START-OF-SELECTION.

  TRY.
      ycl_github_deployer=>deploy(
            repository  = git_repo
            branch      = branch
            github_user = username
            password    = password
            transport   = tranport
            test_only   = test ).
    CATCH cx_root INTO exc.
      MESSAGE exc TYPE 'I' DISPLAY LIKE 'E'.
  ENDTRY.

The "TEST" flag will build the complete archive but, rather than deploying it, will allow you to save it to your local file system.

Continuous Deployment

If you want to implement a Continuous Deployment solution so that the SAPUI5 app is automatically deployed when changes are committed I suggest the following two methods.

If your infratructure guys will allow HTTP calls from outside your corporate network to be passed through to your development WAS you can build a ICF handler that can be called using a GitHub Webhook whenever there is a commit or branch merge to master. When triggered this handler should check the API call for validity before calling the YCL_GITHUB_DEPLOYER class. In my opinion this is the best way to trigger deployment as the code is only executed when actually required.

If you are unable to use the GitHub Webhook method you can build a program that calls the GitHub API to see if the SHA-1 hash of the master branch has changed. If it has then the YCL_GITHUB_DEPLOYER class can be called. I suggest storing the SHA-1 hash in a server-side cookie (CL_BSP_SERVER_SIDE_COOKIE) for this purpose. This program would need to be scheduled as a regular background job so that it regularly checks if a deployment is required.
